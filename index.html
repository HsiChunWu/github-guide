完整教學 · 六階段

GitHub 使用教學指南

從完全新手到進階協作，一頁搞定 GitHub 所有核心技能。

第一階段

入門基礎

1

什麼是 GitHub？版本控制概念簡介

🕰️ 版本控制是「文件的時光機」

想像你在寫一份重要報告，每次改完都另存新檔：報告_v1.docx、報告_v2_final.docx……這個過程很煩，萬一想回頭看某個版本，根本不知道改了什麼。

版本控制就是解決這個問題的工具。它自動記錄每一次變更，讓你可以隨時回到任何歷史版本，也能清楚看到每次改了什麼、是誰改的。

Git vs GitHub — 差在哪裡？

  • Git：安裝在你電腦上的版本控制工具，負責在本地追蹤所有變更。
  • GitHub：一個雲端平台（網站），讓你把 Git 管理的紀錄上傳到網路，方便備份與協作。

簡單說：Git 是引擎，GitHub 是停車場。

為什麼要用 GitHub？

  • 📦 個人備份：電腦壞掉也不怕，程式碼安全在雲端。
  • 💼 找工作：GitHub 是工程師的履歷，面試官常看你的 repo。
  • 🤝 多人協作：多人同時修改不同地方，自動合併，不互相覆蓋。
  • 🌍 開源貢獻：參與全球開源專案，學習與回饋。
2

註冊帳號與介面導覽

① 註冊步驟

  1. 前往 github.com，點右上角「Sign up」。
  2. 輸入電子郵件、設定密碼。
  3. 填入 Username（帳號名稱）。
  4. 通過驗證碼，收取驗證信並點擊連結完成驗證。
  5. GitHub 的問卷（幾人團隊、用途等）可直接按「Skip」略過。
💡
帳號名稱建議：帳號會出現在所有 URL 中（github.com/你的帳號），選個好記又專業的名稱。 ✅ 英文、數字、連字號（-）；❌ 底線、特殊符號、中文。

② 主要介面區域

  • 🔍 搜尋列（頂部）：搜尋 Repository、使用者、程式碼。
  • ➕ 「+」按鈕：快速建立新 Repository、Gist 等。
  • 🔔 鈴鐺圖示：通知中心，顯示有人在你的 PR/Issue 留言等。
  • 👤 頭像選單：個人頁面、設定（Settings）、登出。
  • 📋 左側 Repository 列表：你的專案列表。
3

建立第一個 Repository

什麼是 Repository？

Repository 就像是一個專案的資料夾，但它不只存放檔案，還記錄了從誕生到現在每一次的變更歷史，可以隨時回頭查看任何版本。

建立步驟

  1. 點右上角的 「+」 → 選「New repository」。
  2. 填入 Repository name（例如 my-first-repo）。
  3. 選擇 Public 或 Private。
  4. 勾選 「Add a README file」（強烈建議）。
  5. 視需要選擇 .gitignore 和 License。
  6. 點 「Create repository」 完成！
類型誰能看？適合用途
Public所有人開源專案、個人作品集
Private只有你和邀請的人私人專案、工作用途

README / .gitignore / License 用途

  • 📄 README.md：進入 repo 首先看到的說明書。
  • 🚫 .gitignore：告訴 Git「這些檔案不要追蹤」，例如密碼、暫存檔、node_modules。
  • ⚖️ License：聲明別人可以如何使用你的程式碼（MIT 最寬鬆，個人學習可不選）。
4

認識重要名詞

先把這些名詞搞清楚，後面的操作就會順很多。

# 安裝完成後開啟「Git Bash」來執行 Git 指令

基本 Git 設定

每次做 commit，Git 都會記錄作者的名字和信箱，需要先告訴它你是誰。

git config --global user.name "你的名字"
git config --global user.email "你的信箱@example.com"

--global 表示套用到這台電腦上所有的 Git repo，不需要每個 repo 都重設。

Terminal
$ git config --global user.name "Xiao Ming"
$ git config --global user.email "xiao@example.com"
$ git config --list
user.name=Xiao Ming
user.email=xiao@example.com

第一階段常見問題

Git 和 GitHub 是同一家公司嗎？
不是。Git 是由 Linux 之父 Linus Torvalds 在 2005 年開發的開源工具；GitHub 是 2008 年成立的公司，2018 年被 Microsoft 收購。兩者獨立存在，你也可以用 Git 搭配 GitLab、Bitbucket 等其他平台。
一定要用命令列嗎？有沒有圖形介面？
不一定。有很多 Git GUI 工具，例如 GitHub Desktop（免費，官方出品）、Sourcetree、GitKraken。但學會命令列指令能讓你理解底層邏輯，遇到問題更容易排查，建議先學命令列再用 GUI。
Private repo 要付費嗎？
不用！GitHub 免費帳號可以建立無限個 Private repo，也可以邀請 3 位協作者免費使用。付費的 GitHub Pro / Team 提供更多進階功能，但對個人或小團隊來說，免費版就很夠用了。
帳號名稱之後可以改嗎？
可以，在 Settings → Account → Change username 更改。但改名後，原本的 URL 會變，如果別人有書籤或你的 repo URL 有被引用，需要注意更新。GitHub 會對舊 URL 做短暫的重新導向。
git config --global 和 --local 差在哪？
--global 設定套用到你這台電腦的所有 repo；--local 只套用到目前這個 repo（儲存在 .git/config）。如果你有多個工作帳號，可以在特定 repo 用 --local 覆蓋設定。

第二階段

核心操作

1

Clone 專案到本機

Clone 把 GitHub 上的 repo 完整複製到你的電腦，包含所有歷史紀錄，讓你在本地有一個可操作的完整副本。

操作步驟

  1. 進入 GitHub 上你想 clone 的 repo 頁面。
  2. 點綠色 「<> Code」 按鈕 → 切到 HTTPS 分頁 → 複製 URL。
  3. 打開終端機，切換到想存放的位置，執行 git clone。
Terminal
$ git clone https://github.com/username/my-project.git
Cloning into 'my-project'...
remote: Enumerating objects: 23, done.
Receiving objects: 100% (23/23), 5.12 KiB | 1.02 MiB/s, done.
$ cd my-project && ls -a
.git    README.md    index.html    style.css
my-project/
├── .git/        ← Git 的「大腦」，不要手動修改！
├── README.md
└── index.html
💡
.git/ 是隱藏資料夾，存放所有版本歷史。刪除它整個 repo 就壞了。
2

新增、修改、刪除檔案

在工作目錄新增、修改、刪除檔案，Git 都會注意到，但不會自動存檔。你需要主動告訴 Git 要記錄哪些變更。

三種檔案狀態

📄
Untracked
新建，Git 還不認識
→git add
📋
Staged
放入暫存區，待確認
→git commit
✅
Committed
存入歷史，可回溯
  • Untracked：新建的檔案，Git 還沒開始追蹤。
  • Modified：已追蹤的檔案被改動，尚未準備 commit。
  • Staged：用 git add 放進暫存區，下次 commit 會包含這些。
ℹ️
為什麼需要 Staged？ 因為你可能改了很多檔案，但只想把其中幾個放進這次 commit，暫存區讓你精確控制。
3

git add / git commit / git push 工作流程

  • 🧺 git add：把指定檔案放進暫存區。
  • 💾 git commit：把暫存區內容確認存成快照，需要寫 message。
  • ☁️ git push：把所有未上傳的 commits 同步到 GitHub。
Terminal — 完整 commit 流程
$ git add README.md
# 或用 git add . 加入所有改動
$ git commit -m "更新 README：加入使用說明"
[main 3f2a1c9] 更新 README：加入使用說明
 1 file changed, 5 insertions(+), 1 deletion(-)
$ git push origin main
To https://github.com/username/my-project.git
   a1b2c3d..3f2a1c9  main -> main
💡
Commit message 寫作建議： 動詞開頭（「新增」「修正」「更新」）、說明做了什麼。避免「更新」、「fix」等過於模糊的描述。
4

查看歷史紀錄

  • git status：查看目前工作目錄狀態。
  • git log：完整 commit 歷史（含作者、日期、hash）。
  • git log --oneline：每筆 commit 一行的精簡版。
  • git diff：查看具體改了哪些行。
Terminal
$ git log --oneline
3f2a1c9 更新 README：加入使用說明
a1b2c3d 新增 index.html
8e4d5f2 初始化專案
5

在 GitHub 網頁上直接編輯

不需要開終端機，GitHub 網頁本身就有編輯功能，適合快速修改單一文字檔案。

  1. 進入 repo，點開想編輯的檔案。
  2. 點右上角 鉛筆圖示（✏️ Edit this file）。
  3. 直接修改內容，完成後點 「Commit changes...」。
  4. 填寫 commit message → 確認送出。
ℹ️
適合： 修改 README、修正小錯字、更新連結。
不適合： 需要同時改多個檔案、需要本機測試的變更。
💡
在任何 repo 頁面按下鍵盤 .（英文句點），會開啟 github.dev——完整的 VS Code 網頁版，支援多檔案編輯，完全免安裝！
6

上傳圖片與其他資源

方法一：網頁拖曳上傳

進入 repo → 點 「Add file」→「Upload files」 → 拖曳上傳 → 填 commit message → 送出。

方法二：指令上傳

Terminal
$ git add images/cover.png
$ git commit -m "新增封面圖片"
[main 9c8d7e6] 新增封面圖片
$ git push origin main
⚠️
不建議在 repo 放大型二進位檔！ GitHub 單檔限制 100 MB，整個 repo 建議不超過 1 GB。影片、大型資料集請使用 Git LFS 或其他儲存方案。

第二階段常見問題

push 時要求輸入密碼，但密碼輸入正確還是失敗？
GitHub 在 2021 年後不再接受密碼驗證 HTTPS push，需要改用 Personal Access Token（PAT）。到 Settings → Developer settings → Personal access tokens 產生 token，把它當密碼貼入即可。或者改用 SSH 金鑰（見第四階段）。
git add . 和 git add -A 有什麼差別？
在大多數情況下效果相同，都是加入所有有變動的檔案。嚴格說：git add . 只作用於目前目錄及子目錄；git add -A 作用於整個 repo 根目錄。對一般用法來說差異不大，git add . 最常用。
commit 之後發現 message 寫錯了，可以改嗎？
可以，但只適用於還沒 push 的 commit：
git commit --amend -m "正確的訊息"
如果已經 push 了，修改會改寫歷史，需要用 --force，對協作 repo 要特別謹慎。
push 時出現「rejected」是什麼意思？
表示遠端有你本地沒有的 commit（通常是其他人先 push 了）。需要先 git pull 把遠端的變更拉下來、解決可能的衝突，再重新 push。
可以一次 push 多個 commit 嗎？
可以！你在本機做了幾個 commit，執行一次 git push 就會把所有未上傳的 commits 一起推上去。這也是為什麼建議把相關的改動拆成多個 commit，再一起 push。

第三階段

分支與協作

1

什麼是 Branch？為什麼要用？

Branch（分支）讓你從主線「開叉」出去，在一個獨立的環境裡開發新功能或修 bug，不會影響主線的穩定版本。等開發完成並確認沒問題，再合併回主線。

  • 🛡️ 保護主線：main 永遠維持穩定可用狀態。
  • 👥 多人並行：A 開發登入、B 開發首頁，互不干擾。
  • 🔬 安全實驗：在 branch 上大膽嘗試，失敗了直接刪掉。
2

建立與切換分支

Terminal
# 建立新 branch 並立刻切換過去（最常用）
$ git checkout -b feature-login
Switched to a new branch 'feature-login'

# 查看所有 branch（* 表示目前所在）
$ git branch
* feature-login
  main

# 切換到已存在的 branch
$ git checkout main
Switched to branch 'main'

# 刪除已合併的 branch
$ git branch -d feature-login
Deleted branch feature-login (was 3f2a1c9).
💡
新版 Git 也支援 git switch -c feature-login（建立並切換）和 git switch main（切換），語意更清楚，推薦使用。
3

Merge 合併分支

開發完成後，把 feature branch 合併回 main。

Terminal
# 先切回 main
$ git checkout main

# 把 feature-login 合併進來
$ git merge feature-login
Updating a1b2c3d..3f2a1c9
Fast-forward
 login.html | 45 ++++++++++++
 1 file changed, 45 insertions(+)
ℹ️
Fast-forward：main 沒有新 commit，Git 直接把指針移過去，最乾淨的合併方式。
4

衝突（Conflict）發生時怎麼處理

兩個 branch 都修改了同一個檔案的同一行，合併時就會產生衝突。Git 無法自動判斷該保留哪個版本，需要你手動解決。

<<<<<<< HEAD
這是 main 的內容
=======
這是 feature-login 的內容
>>>>>>> feature-login

解決步驟

  1. 打開有衝突的檔案，找到 <<<<<<< 標記。
  2. 決定要保留哪段內容（或兩者都保留），刪除 Git 加的標記符號。
  3. 儲存檔案，執行 git add → git commit 完成合併。
💡
VS Code 遇到衝突會顯示「Accept Current Change / Accept Incoming Change / Accept Both」按鈕，點一下就能解決，比手動編輯方便很多。
5

Fork 別人的專案

想修改別人的開源專案，但你沒有權限直接 push？先 Fork！

Fork 會在你的帳號下複製一份完整的 repo，你可以在上面自由修改，再透過 Pull Request 把改動提交給原作者審核。

  1. 進入想要 fork 的 repo 頁面。
  2. 點右上角的 「Fork」 按鈕。
  3. 選擇要 fork 到哪個帳號，點 「Create fork」。
  4. 現在你帳號下有了一份副本，clone 它到本機，隨意修改。
ℹ️
Fork 和 Clone 的差別：Fork 是在 GitHub 上建立副本（存在雲端）；Clone 是把 repo 下載到本機。通常兩者一起用：先 Fork，再 Clone 自己的 fork。
6

Pull Request（PR）：提交貢獻

Pull Request 是向 repo 提交你的變更，請對方審核後合併。是開源貢獻和團隊協作的核心流程。

建立 PR 的步驟

  1. 在你的 branch（或 fork）上完成修改並 push 到 GitHub。
  2. 進入 GitHub repo 頁面，系統通常會出現「Compare & pull request」黃色提示列，點擊它。
  3. 填寫 PR 標題和描述（說明改了什麼、為什麼）。
  4. 點 「Create pull request」 送出。
  5. 等待對方 review → 修改 → 對方按 「Merge pull request」。
💡
好的 PR 描述應包含： 做了什麼改動、為什麼這樣改、如何測試、有沒有相關 Issue。讓 reviewer 能快速理解你的意圖。
7

Issue：回報問題與討論

Issue 是 GitHub 內建的「問題追蹤系統」，可以用來：回報 bug、提出新功能需求、討論技術問題、追蹤待辦事項。

建立 Issue

  1. 進入 repo → 點頂部的 「Issues」 標籤。
  2. 點 「New issue」。
  3. 填寫標題（明確描述問題）和內文（重現步驟、錯誤訊息、期望行為）。
  4. 可以指定 Assignees（負責人）、Labels（分類標籤）、Milestone。
  5. 送出後，相關的 PR 可以在 message 中寫 Closes #123 讓合併時自動關閉 issue。
💡
在 commit message 或 PR 中寫 Closes #42，PR 被 merge 後 Issue #42 會自動關閉。

第三階段常見問題

main 和 master 有什麼差別？
GitHub 在 2020 年把新 repo 的預設主分支從 master 改名為 main。兩者功能完全相同，只是名稱不同。舊的 repo 可能仍叫 master。可用 git branch -M main 重新命名。
Fork 之後，原 repo 更新了，我的 fork 會自動同步嗎？
不會。你需要手動同步：在 GitHub 網頁上，你的 fork 頁面會顯示「Sync fork」按鈕，點擊即可把原 repo 的最新變更拉進來。或用指令設定 upstream remote 再 pull。
Branch 刪掉後，上面的 commit 還在嗎？
如果已經 merge，那些 commit 已經進入主線，刪 branch 只是刪掉「指針」，不影響 commit 歷史。如果還沒 merge 就刪 branch，commit 會暫時變成孤立狀態，Git 會在一段時間後清理。可用 git reflog 找回。
一個 PR 可以對應多個 commit 嗎？
可以！PR 其實是把一整個 branch 的所有 commit 請求合併，branch 上可以有一個或多個 commit，都會一起出現在 PR 的 Files changed 頁面供 reviewer 審查。
衝突很可怕，有辦法避免嗎？
可以減少但無法完全避免。建議：① 頻繁從 main pull 最新版到你的 branch（git pull origin main）；② 每個人負責不同的檔案或模組；③ 保持 branch 存活時間短，盡快 PR 合併。

git revert vs git reset：撤銷變更

做錯了想反悔，有兩種方式，適用情境不同：

指令效果適用時機安全性
git revert <hash> 建立一個新 commit 來「反做」指定的 commit 已 push 到遠端、多人協作的 repo ✅ 安全，不改寫歷史
git reset --soft HEAD~1 取消最近一次 commit，但保留改動在暫存區 還沒 push，想修改後重新 commit ⚠️ 只在本機用
git reset --hard HEAD~1 完全丟棄最近一次 commit 及所有改動 確定不需要那些改動，想徹底清除 ❌ 危險，無法還原
Terminal
# 安全地反悔某個 commit（不改寫歷史）
$ git revert 3f2a1c9
[main 8a3b2c1] Revert "更新 README：加入使用說明"

# 取消最近一次 commit，保留改動
$ git reset --soft HEAD~1

# 查看 commit hash（用於 revert）
$ git log --oneline
3f2a1c9 更新 README：加入使用說明
a1b2c3d 新增 index.html
🚨
git reset --hard 無法還原！ 執行前請三思。如果已經 push 過，強制改寫歷史需要 git push --force，會造成其他協作者的困擾。

Tag 與 Release：標記版本

Tag 讓你在某個重要的 commit 上貼上「書籤」，通常用來標記軟體版本（如 v1.0.0）。Release 是 GitHub 在 tag 基礎上加的說明頁面，可以附帶下載檔案。

建立 Tag

Terminal
# 建立帶說明的 annotated tag（推薦）
$ git tag -a v1.0.0 -m "第一個正式發布版本"

# 把 tag 推送到 GitHub
$ git push origin v1.0.0
To https://github.com/username/my-project.git

# 查看所有 tag
$ git tag
v0.9.0
v1.0.0

建立 Release

在 GitHub 網頁上：repo 頁面右側 → 「Releases」→「Create a new release」 → 選擇 tag → 填寫版本說明 → 可上傳安裝包 → 點 「Publish release」。

💡
版本號建議遵循 語意化版本（Semantic Versioning）：MAJOR.MINOR.PATCH。例如 v2.1.3 表示第 2 個主版本、第 1 個新功能、第 3 個 bug 修復。

GitHub Actions 入門：自動化 CI/CD

GitHub Actions 讓你在特定事件（push、PR、定時）發生時，自動執行工作流程，例如：跑測試、打包、部署到伺服器。設定方式是在 repo 裡建立 .github/workflows/ 目錄，放入 YAML 設定檔。

一個簡單的自動測試範例

# .github/workflows/test.yml
name: Run Tests

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Install dependencies
        run: npm install
      - name: Run tests
        run: npm test

把這個檔案推上去後，每次 push 到 main 或建立 PR，GitHub 就會自動在雲端跑測試，並在 PR 頁面顯示通過 ✅ 或失敗 ❌。

ℹ️
GitHub Actions 對 Public repo 完全免費，Private repo 每月有 2,000 分鐘免費額度（免費帳號），通常對個人專案足夠。

SSH 金鑰設定：免密碼登入

每次 push 都要輸入 token 很煩？設定 SSH 金鑰後，push/pull 都不需要輸入密碼，更安全也更方便。

設定步驟

Terminal
# 1. 產生 SSH 金鑰（一路按 Enter 使用預設值即可）
$ ssh-keygen -t ed25519 -C "你的信箱@example.com"
Generating public/private ed25519 key pair.
Enter file in which to save the key (~/.ssh/id_ed25519):
Your identification has been saved in ~/.ssh/id_ed25519

# 2. 複製公鑰內容
$ cat ~/.ssh/id_ed25519.pub
ssh-ed25519 AAAA... 你的信箱@example.com

# 3. 測試連線
$ ssh -T git@github.com
Hi username! You've successfully authenticated.

把公鑰加到 GitHub

  1. 複製 cat ~/.ssh/id_ed25519.pub 的輸出內容。
  2. 到 GitHub → Settings → SSH and GPG keys → 「New SSH key」。
  3. 貼上公鑰內容，填入名稱（例如「My MacBook」），點 「Add SSH key」。
  4. 之後 clone 時改用 SSH 網址（git@github.com:username/repo.git）即可免密碼操作。

多人協作最佳實踐

掌握了工具之後，協作流程的品質取決於團隊的工作習慣。以下是經過驗證的最佳實踐：

🌿 Branch 策略

  • main 永遠保持可部署狀態，不直接 commit。
  • 每個功能或 bug 修復開一個獨立 branch（命名如 feature/login、fix/typo-in-readme）。
  • Branch 壽命要短，完成後盡快合併並刪除。

📝 Commit 習慣

  • 每個 commit 只做一件事，不要把不相關的改動混在一起。
  • Commit message 清楚說明做了什麼（動詞開頭）。
  • 不要 commit 未完成的程式碼到 main。

🔍 Code Review

  • 所有改動透過 PR 進行，至少一人 review 才能合併。
  • PR 盡量小，讓 review 更容易；大功能拆成多個小 PR。
  • review 時提供建設性意見，而不只是「LGTM」。

🔄 同步習慣

  • 每天開始工作前先 git pull，保持本機最新。
  • 長壽 branch 要頻繁從 main 同步（git merge main 或 git rebase main）。
💡
保護 main branch： 到 repo Settings → Branches → Add branch protection rule，可設定「需要 PR 才能合併」、「需要至少 N 個 review」等規則，防止直接 push 到 main。

第四階段常見問題

git revert 和 git reset 我該用哪個？
簡單判斷：已 push 到遠端 → 用 revert（安全，不改寫歷史）；還沒 push → 可以用 reset（更乾淨，但只在本機用）。在多人協作的 repo 上絕對不要對已 push 的 commit 用 reset --hard 再 force push，會造成其他人的困擾。
GitHub Actions 免費額度用完了怎麼辦？
Public repo 的 Actions 不計入額度，完全免費。Private repo 的 Actions 用完免費額度後，可以在 Settings → Billing 升級方案，或購買額外的分鐘數。也可以考慮把 CI 工作搬到其他免費服務，如 CircleCI、GitLab CI 等。
SSH 金鑰丟了或電腦換了怎麼辦？
到 GitHub Settings → SSH keys 刪除舊金鑰，在新電腦重新產生一組金鑰並加入。舊的連線就會失效，只有新金鑰能用。建議把金鑰名稱標注設備名稱（如「Work MacBook 2024」），方便管理。
什麼是 git rebase？和 merge 有什麼不同？
Rebase 會把你的 commits「搬移」到目標 branch 的最新點上，讓歷史看起來是線性的，沒有合併節點。Merge 保留分叉歷史，更真實但較雜亂。Rebase 讓 log 更整潔，但會改寫 commit 歷史，不建議在已 push 的 branch 上使用。
tag 和 branch 有什麼根本差別？
Branch 是會移動的指針，隨著新 commit 不斷前進。Tag 是固定的書籤，永遠指向同一個 commit，通常用來標記版本號。建立 tag 後，除非手動刪除再建，它不會移動。

第五階段常見問題

GitHub Pages 啟用後要等多久才能瀏覽？
通常 1～3 分鐘。若超過 5 分鐘還沒上線，可以到 repo 的 Actions 標籤查看 deployment 狀態，看是否有錯誤。常見原因是 index.html 不在 root 目錄，或 branch 選錯了。
push 後 GitHub Pages 沒更新怎麼辦？
GitHub Pages 更新需要 1～2 分鐘，請稍等後按 Ctrl+Shift+R（硬重新整理）清除瀏覽器快取。若仍未更新，到 Settings → Pages 確認設定正確，或到 Actions 標籤查看部署記錄是否有失敗。
可以在 GitHub Pages 用自訂網域嗎？
可以！在 Settings → Pages 的「Custom domain」欄填入你的網域（例如 notes.example.com），然後到你的 DNS 服務商設定 CNAME 指向 你的帳號.github.io。GitHub 也支援免費的 HTTPS 自動憑證。
Merge 自己的 PR 和直接 push 到 main 有什麼差別？
功能上結果相同，但流程和習慣不同。透過 PR merge 可以保留完整的開發記錄（誰改了什麼、為什麼），在協作專案中讓其他人有機會 review。直接 push 較快，適合個人小修改。建議養成用 PR 的習慣，未來加入團隊時能直接適應。
做壞了，怎麼用 git revert 回復？
找到你想撤銷的 commit hash（用 git log --oneline），然後執行 git revert <hash>。這會建立一個新的 commit 來「反做」那次變更，不會改寫歷史，push 後 GitHub Pages 也會自動更新回去。

Git 指令速查

1

設定與初始化：git config、git init

git config — 設定使用者資訊

在開始使用 Git 之前，需先設定名稱與 Email，這些資訊會出現在每一個 commit 記錄中。

bash
$ git config --global user.name "你的名字"
$ git config --global user.email "email@example.com"
$ git config --global core.editor "code --wait"  # 設定編輯器為 VS Code
$ git config --list                                # 列出所有生效設定
參數說明
--global套用到目前使用者的所有 repo（最常用）
--local只套用到目前 repo（預設，優先權最高）
--system套用到系統所有使用者
--list列出所有生效的設定值

git init — 建立新 Repo

bash
$ git init                   # 在目前目錄初始化
$ git init my-project        # 建立新資料夾並初始化
$ git init --bare server.git # 建立裸 repo（用於伺服器）
💡
設定優先順序：--local > --global > --system。可為不同專案設定不同 Email，例如工作專案用公司帳號，個人專案用私人帳號。
2

日常操作：status、add、commit

git status — 查看目前狀態

bash
$ git status      # 詳細狀態
$ git status -s   # 簡短格式：M=修改, ??=未追蹤
$ git status -u   # 顯示所有未追蹤的檔案

git add — 加入暫存區

bash
$ git add README.md  # 加入特定檔案
$ git add src/        # 加入整個目錄
$ git add .           # 加入所有變更
$ git add -p          # 互動式選取部分修改（patch）
$ git add -u          # 只加入已追蹤的修改，略過新增檔案
參數說明
-p / --patch逐段（hunk）選取要暫存的內容，同一個檔案可以只加部分修改
-u / --update更新所有已追蹤的檔案，略過新增的未追蹤檔案
-A / --all包含新增、修改、刪除，等同 git add .

git commit — 提交

bash
$ git commit -m "feat: 新增登入功能"
$ git commit -am "fix: 修正 null 錯誤"   # add + commit（只對已追蹤檔案）
$ git commit --amend                      # 修改最後一個 commit
$ git commit --amend --no-edit            # 修改內容但保留原訊息
⚠️
--amend 會改寫歷史，若已 push 到遠端請避免使用，否則需要 force push，會影響其他協作者。
3

分支管理：branch、switch、merge、rebase

git branch — 管理分支

bash
$ git branch                    # 列出本地分支（* 標示目前）
$ git branch -a                 # 列出本地與遠端分支
$ git branch feature/login     # 建立新分支（不切換）
$ git branch -d feature/login  # 刪除分支（需已合併）
$ git branch -D feature/login  # 強制刪除（未合併也可）
$ git branch -m old new        # 重新命名分支

git switch — 切換分支（Git 2.23+ 新語法）

bash
$ git switch main              # 切換到 main
$ git switch -c feature/login  # 建立並切換（舊語法：checkout -b）
$ git switch -                 # 切回上一個分支

git merge — 合併分支

bash
$ git merge feature/login           # 合併到目前分支
$ git merge --no-ff feature/login   # 強制建立 merge commit
$ git merge --squash feature/login  # 壓縮成一個 commit
$ git merge --abort                 # 有衝突時放棄合併
merge 參數說明
--no-ff保留分支歷史，即使可以 fast-forward 也建立 merge commit
--squash把分支所有 commit 壓縮成一個，需手動再 commit
--abort遇到衝突時取消整個合併

git rebase — 變基

將目前分支的 commit「搬移」到另一個基礎點，讓歷史更線性。

bash
$ git rebase main        # 把目前分支接到 main 最新點
$ git rebase -i HEAD~3   # 互動式整理最近 3 個 commit
$ git rebase --abort     # 中止
$ git rebase --continue  # 解完衝突後繼續
互動模式（-i）指令說明
pick保留此 commit
reword修改 commit 訊息
squash與上一個 commit 合併
drop刪除此 commit
🚨
rebase 會改寫 commit 歷史。若分支已推送到遠端並有其他人使用，請勿對其執行 rebase，否則會造成協作者的歷史衝突。
4

遠端操作：remote、fetch、pull、push

git remote — 管理遠端連結

bash
$ git remote -v                              # 列出所有遠端與 URL
$ git remote add origin https://github.com/…  # 新增遠端
$ git remote remove origin                    # 移除遠端
$ git remote rename origin upstream           # 重新命名
$ git remote set-url origin https://…         # 更改 URL

git fetch — 抓取但不合併

bash
$ git fetch          # 抓取預設遠端（origin）
$ git fetch --all    # 抓取所有遠端
$ git fetch --prune  # 刪除本地已不存在於遠端的追蹤分支

git pull — fetch + merge

bash
$ git pull               # 等同 fetch + merge
$ git pull --rebase      # 用 rebase 代替 merge，歷史更線性
$ git pull origin main  # 明確指定遠端與分支

git push — 推送到遠端

bash
$ git push origin main               # 推送 main 分支
$ git push -u origin feature/login   # 推送並設定追蹤（之後可直接 git push）
$ git push --force-with-lease        # 安全的強制推送
$ git push origin --delete old-branch  # 刪除遠端分支
$ git push --tags                    # 推送所有 tag
💡
強制推送請用 --force-with-lease 而非 -f。前者在推送前會確認遠端沒有你未知的新 commit，避免覆蓋他人工作。

查看歷史：log、diff、show、blame

git log — 瀏覽 commit 記錄

bash
$ git log --oneline               # 每個 commit 一行
$ git log --oneline --graph --all  # 所有分支的圖形化歷史
$ git log --author="姓名"         # 篩選作者
$ git log --since="2 weeks ago"   # 篩選時間
$ git log -p                      # 顯示每個 commit 的 diff
$ git log -- path/to/file         # 只看某個檔案的歷史

git diff — 比較差異

bash
$ git diff                  # 工作目錄 vs 暫存區
$ git diff --staged         # 暫存區 vs 最後一個 commit
$ git diff main feature/x  # 兩個分支之間的差異
$ git diff HEAD~2           # 與兩個 commit 前比較

git show 與 git blame

bash
$ git show abc1234              # 查看特定 commit 的詳細內容
$ git show HEAD~1               # 查看上一個 commit
$ git blame README.md           # 查看每行是誰、何時修改
$ git blame -L 10,20 README.md  # 只看第 10~20 行
ℹ️
查詢完整說明可直接在終端機執行 git help <指令名稱>（例如 git help log），或加 --help 旗標（例如 git log --help）。

復原操作：restore、reset、revert

git restore — 還原工作目錄或暫存區（Git 2.23+）

bash
$ git restore README.md           # 丟棄工作目錄的修改
$ git restore --staged README.md  # 從暫存區移除（unstage）

git reset — 移動 HEAD

bash
$ git reset HEAD~1         # 撤銷 commit，修改回工作目錄（--mixed 預設）
$ git reset --soft HEAD~1  # 撤銷 commit，修改保留在暫存區
$ git reset --hard HEAD~1  # 撤銷 commit，丟棄所有修改（危險！）
模式HEAD暫存區工作目錄
--soft移動不變不變
--mixed（預設）移動清除不變
--hard移動清除清除

git revert — 安全撤銷（保留歷史）

revert 不改寫歷史，而是建立一個「反向 commit」來抵消舊的改動，適合已推送到遠端的情況。

bash
$ git revert HEAD           # 撤銷最後一個 commit
$ git revert abc1234        # 撤銷特定 commit
$ git revert HEAD~3..HEAD   # 撤銷最近 3 個 commit
⚠️
reset 還是 revert？ 未推送到遠端 → 可用 reset（--hard 請謹慎）。已推送到遠端 → 使用 revert，不改寫共享歷史，不影響其他協作者。

其他實用指令：stash、tag、cherry-pick、bisect

git stash — 暫存未完成的修改

需要緊急切換分支處理 bug，又不想 commit 當前未完成工作時使用。

bash
$ git stash                    # 儲存工作目錄與暫存區
$ git stash push -m "WIP 登入"  # 附說明
$ git stash list               # 列出所有 stash
$ git stash pop                # 取出最新 stash 並刪除
$ git stash apply stash@{1}    # 取出特定 stash（保留 stash）
$ git stash drop stash@{0}     # 刪除特定 stash
$ git stash clear              # 清除所有 stash

git tag — 標記版本

bash
$ git tag v1.0.0                   # 輕量 tag
$ git tag -a v1.0.0 -m "正式版"    # 附注 tag（推薦，含作者與時間）
$ git tag                           # 列出所有 tag
$ git push origin v1.0.0           # 推送特定 tag
$ git push --tags                   # 推送所有 tag

git cherry-pick — 複製特定 commit

把某個分支上的特定 commit 複製到目前分支，不需要合併整個分支。

bash
$ git cherry-pick abc1234           # 複製一個 commit
$ git cherry-pick abc1234..def5678  # 複製一段範圍
$ git cherry-pick -n abc1234        # 複製但不自動 commit

git bisect — 二分搜尋找 bug

不知道哪個 commit 引入了 bug 時，讓 Git 自動用二分法縮小範圍。

bash
$ git bisect start
$ git bisect bad            # 目前版本有 bug
$ git bisect good v1.0.0   # 這個版本正常
# Git 自動切到中間 commit，測試後繼續標記：
$ git bisect good   # 或 git bisect bad
$ git bisect reset          # 找到後結束，回到原本 HEAD

第六階段常見問題

git fetch 和 git pull 有什麼不同？
git fetch 只把遠端的最新資料抓回本機，不會改動工作目錄和分支。git pull 則是 fetch + merge（或 rebase），會直接把遠端變更合併進來。建議先 fetch 後用 git diff origin/main 確認差異，再決定是否 merge。
什麼時候用 merge，什麼時候用 rebase？
merge 保留所有分支歷史，適合把功能分支合入 main，讓大家都能看到「這個功能什麼時候被合入」。rebase 整理歷史讓它更線性，適合在個人功能分支上「把 main 的最新更新接進來」，讓 PR 更容易審查。黃金法則：已推送到共享遠端的分支，不要對其 rebase。
不小心 git reset --hard，修改都不見了怎麼辦？
先別慌！Git 的 reflog 記錄了所有 HEAD 的移動歷史，只要沒超過約 90 天，通常都能救回：執行 git reflog 找到 reset 前的 commit hash，再執行 git reset --hard <hash> 即可還原。這是 Git 最強大的安全網之一。
git stash pop 和 git stash apply 差在哪？
git stash pop 取出最新的 stash 並從清單中刪除它。git stash apply 取出內容但保留 stash，可以重複套用到不同分支。通常用 pop；如果需要把相同修改套用到多個地方才用 apply。
